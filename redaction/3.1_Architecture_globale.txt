3.1 Architecture globale du système

Cette section présente l'architecture complète du système de détection d'armes à feu sécurisé contre les attaques adverses. Nous décrivons les composants principaux, leurs interactions et le flux de données depuis l'acquisition des données jusqu'au déploiement en production.

================================================================================
3.1.1 Vue d'ensemble de l'architecture
================================================================================

Notre système est conçu selon une architecture modulaire composée de quatre couches principales :

1. **Couche de données** : Gestion du dataset personnalisé (images d'armes à feu et images sûres)
2. **Couche d'entraînement** : Pipeline d'entraînement adversarial avec MobileNetV2
3. **Couche de services** : API REST pour l'inférence et l'évaluation
4. **Couche de présentation** : Interface web pour l'interaction utilisateur

Cette architecture permet une séparation claire des responsabilités et facilite la maintenance, les tests et le déploiement du système.

[INSÉRER FIGURE: Schéma architecture globale - Voir fichier séparé: diagramme_architecture_globale.txt]

Les principales caractéristiques de cette architecture incluent :

- **Modularité** : Chaque composant peut être développé et testé indépendamment
- **Scalabilité** : L'utilisation de Docker permet un déploiement flexible et scalable
- **Sécurité** : L'entraînement adversarial est intégré nativement dans le pipeline
- **Reproductibilité** : Tous les hyperparamètres et configurations sont versionnés


================================================================================
3.1.2 Diagramme de composants
================================================================================

Le système est composé de plusieurs modules interconnectés, chacun ayant une responsabilité spécifique dans le processus global de détection sécurisée.

[INSÉRER FIGURE: Diagramme de composants - Voir fichier séparé: diagramme_composants.txt]

**A. Composants de la couche de données**

- **Dataset Loader** : Charge et prétraite les images d'armes à feu (Kaggle) et images sûres (COCO)
- **Data Augmentation** : Applique des transformations aléatoires (rotation, zoom, flip, etc.)
- **Data Validator** : Vérifie l'intégrité et la qualité des données

**B. Composants du pipeline d'entraînement**

- **MobileNetV2 Base Model** : Architecture CNN légère et performante
- **Adversarial Training Module** : Génère des exemples adverses (FGSM, PGD) durant l'entraînement
- **Training Manager** : Coordonne l'entraînement standard et adversarial
- **Metrics Tracker** : Enregistre les métriques (accuracy, loss, robustesse)
- **Model Checkpointing** : Sauvegarde les meilleurs modèles

**C. Composants de déploiement**

- **Docker Container** : Encapsule l'environnement d'exécution complet
- **FastAPI/Flask Server** : Expose les endpoints REST pour l'inférence
- **Model Serving** : Charge et gère les modèles entraînés
- **Web Interface** : Interface utilisateur pour tester le système

**D. Composants d'évaluation**

- **Robustness Evaluator** : Teste le modèle contre diverses attaques adverses
- **Performance Monitor** : Mesure la latence et le throughput
- **Visualization Tools** : Génère des graphiques et rapports


================================================================================
3.1.3 Flux de données et présentation du dataset
================================================================================

**A. Flux de données du système**

Le flux de données dans notre système suit quatre phases principales :

[INSÉRER FIGURE: Diagramme de flux de données - Voir fichier séparé: diagramme_flux_donnees.txt]

**Phase 1 - Préparation des données**
1. Chargement des images d'armes à feu depuis le dataset Kaggle
2. Chargement des images sûres depuis le dataset COCO
3. Normalisation et redimensionnement à 224×224 pixels
4. Séparation en ensembles d'entraînement (70%), validation (15%) et test (15%)
5. Application d'augmentation de données pour l'ensemble d'entraînement

**Phase 2 - Entraînement**
1. Entraînement standard sur données propres (baseline)
2. Génération d'exemples adverses à chaque batch
3. Entraînement adversarial (mélange données propres + adverses)
4. Validation sur données propres et adverses
5. Sauvegarde du meilleur modèle

**Phase 3 - Évaluation**
1. Test sur données propres (précision standard)
2. Génération d'attaques FGSM, PGD, C&W
3. Test de robustesse contre ces attaques
4. Comparaison baseline vs modèle sécurisé
5. Génération de rapports et visualisations

**Phase 4 - Déploiement et Prédiction**
1. Chargement du modèle dans le conteneur Docker
2. Réception d'une image via l'API REST
3. Prétraitement de l'image
4. Inférence avec le modèle sécurisé
5. Retour de la prédiction (arme à feu détectée ou non)

**Justification du choix du dataset :**
Le choix de combiner des images d'armes à feu de Kaggle avec des images sûres de COCO permet :
- **Diversité** : Large variété de contextes et scénarios réalistes
- **Qualité** : Datasets reconnus et validés par la communauté
- **Équilibre** : Possibilité de balancer les classes pour éviter le biais
- **Généralisation** : Meilleure capacité du modèle à généraliser sur de nouvelles images
